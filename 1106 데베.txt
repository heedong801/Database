관계 제약 조건
=========================================================
관계형 데이터 모델에서 지켜져야 하는 무결성 조건 : 관계 제약 조건
반드시 지켜져야 하는 조건
DBMS에서 시스템적으로 보장을 해주어야 함(체크를 해줘야 됨) : 읽기연산은 체크할 필요 없음

사용자들은 지키기 힘듬 사용자가 지키는 건 아니라는 소리
<키 제약조건, 도메인 제약조건, 엔티티무결성 제약조건, 참조무결성 제약조건> : 암기 !

도메인 제약조건
: 
키 제약조건
: 기본 키 Attribute에 중복된 값이 있으면 안된다.
=========================================================
CandidateKey : 뺄수 없는 Attribute의 조합 (하나라도 빠지면 키가 될 수 없음), 하나의 엔티티에 여러 개의 Candidate Key가 존재할 수 있음
PrimaryKey : 여러 개의 Candidate Key에서 하나 고른 것
SuperKey 에서 Candidate Key를 골라냄

관계형 데이터 읽는 법
-두 개의 데이터가 나뉘어져 있지만, 공통 데이터를 통해 합쳐서 읽을 수 있다.

기본키를 구성하는 애트리뷰트는 NULL값을 가질 수 있을까?
-기본키 attribute 는 Search의 시발점이 되는 도구이기 때문에 NULL이 될 수 없다. = 엔티티 무결성 제약조건

참조 무결성 제약조건
: 어떤 Relations에 있는 외래키의 값은얘가 참조하는 ㅌ
=========================================================
ex) D_Num은 Department에 NUMBERT를 참조한다. 참조 관계에 놓여있다.
ex) 순환관계타입 S_ssn은 자신의 ssn을 참조하고 있다.
외래키 : 나는 여기서 키는 아니지만 원래 있는 데로 가면 키임 ( 데이터의 연관성을 나타냄 )
화살표 표시가 되어 있는 부분은 반드시 기본키(머리)

그럼 외래키는 NULL이 들어갈 수 있나?
-가능 (선택적 관계일 때, 즉 참여하지 않음), 의무적 관계의 경우 NULL이 들어올 수 없음

외래키는 중복된 값이 있을 수 있나?
-가능 ex) 1:N 관계타입

근데 중복이 없어야 되는 경우는 ?
-1:1 관계일 때

삽입시 4가지 제약 조건을 모두 위반할 수 있기 때문에 4개의 대해서 다 체크 해주어야 합니다
체크시 삽입 하지 말고 알려줘야 함

삭제
